# Робота та структура автокомпілятора

У компілятор C вбудовано так звану систему автокомпілятора. Значення цього полягає в тому, що коли програма на C викликає будь-яку нову функцію (процедуру), автокомпілятор перевіряє, чи включена дана процедура до будь-якого з включених файлів бібліотек, і якщо він знаходить її в одному з них, то в рамках процесу компіляції цей розділ коду автоматично та виключно включається до вихідного коду, який буде згенеровано. Під час включення автокомпілятор враховує, чи вже включено даний розділ коду до скомпільованого розділу asm (наприклад, на нього вже було посилання в попередньому виклику), і копіює цей розділ у згенерований код лише тоді, коли він ще не включений.
Ця система дозволяє легко компонувати розділи програми, написані на асемблері, у скомпільовану програму. (Вам не потрібно \#включати їх.) Ще одна функція полягає в наданні систематичної допомоги програмісту у вигляді короткої довідки на панелі «Функції автокомпілятора», яку можна запустити натисканням клавіші «F1».

## Структура файлу бібліотеки автокомпонування

Файл бібліотеки автокомпонування — це, по суті, вихідний файл, написаний мовою асемблера, який містить вихідний код частин програми, що підлягають включенню, розширений спеціальними однорядковими інструкціями скрипта. (Я планую пізніше включити частини, написані на C, до вихідного коду.)
Таким чином, ці інструкції скрипта розташовуються на одному рядку та повинні починатися строго на початку рядка. Ще однією їхньою характеристикою є те, що кожна команда скрипта починається з символу '' (крапка з комою), тому компілятор асемблера пропускає їх, оскільки інтерпретує їх як коментарі.

Давайте тепер детально розглянемо ці команди скрипта та одразу почнемо їхню структуру з команд, які повинні бути написані строго в перших рядках (їх можна писати, оскільки вони не є обов'язковими). ​​Це:

|                      |                                                                                                                                          |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `ver: 'version str'` | Наступна частина — це рядок, що описує версію бібліотеки                                                                                 |
| `colb: 'R,G,B'`      | Параметр — це значення кольору R,G,B, розділене комами. Він визначає колір фону в сітці панелі «Функції автоматичного компонування».     |
| `colf: 'R,G,B'`      | Параметр — це значення кольору R, G, B, розділені комами. Він визначає колір шрифту в сітці панелі «Функції автоматичного компонування». |

Приклад:

```
;ver: DB Game Lib (v:1.0.0)
;colb: 210,0,0
;colf: 0,150,150
```

Далі наведено найважливішу команду скрипта, яка має два різні формати:

1) `alnk: 'procedure_name v. block_name' [@ called_procedure1, called_procedure2, ...] [!]`
2) `alnk: 'procedure_name' [,'procedure_name2',.. ] alias 'block_name'`

`;alnk: 'procedure_name v. block_name' [@ called_procedure1, called_procedure2, ...] [!]`

Перша формула складається з префікса 'alnk:' та наступного 'procedure_name' або 'block-name' як обов'язкових елементів. Зазвичай, він позначає початок частини, яка копіюється у згенерований asm-файл під час компіляції. У випадку, коли процедура викликається безпосередньо з C, procedure_name починається з малої літери 'q'.

Однак часто необхідно, щоб, коли компонувальник копіює задану частину коду, він також автоматично зв'язував інші процедури, оскільки, наприклад, наша процедура також викликає інші підпрограми. У цьому випадку імена викликаних процедур повинні бути перелічені після символу '@', розділені комою. (Цей список, що вводиться за допомогою символу '@', звичайно, необов'язковий, його не обов'язково вказувати.) Мітки посилань 'called_procedure1' та 'called_procedure2' тощо НЕ є іменами підпрограм, а іменами процедур або блоків, зазначеними в блоці 'alnk:' заданої процедури!
Є ще один необов'язковий керуючий символ, який необхідно поставити в кінці рядка, якщо він використовується. Це символ знака оклику ('!'). Якщо цей символ присутній в кінці рядка, то під час компонування задана секція коду розміщується безпосередньо перед скомпільованою частиною з вихідного коду C, таким чином гарантуючи, що дана секція коду випадково не буде розміщена під VIDEO RAM на сторінці пам'яті 2. Це означає, що вона дуже рідко використовується.

Приклад1:

```
;alnk: qPeek:
qPeek:
	POP BC
	POP HL ;'HL' = addr
	PUSH HL
	PUSH BC
	LD L, (HL)
	LD H,0
	RET
;#end
```

Приклад2:

```
;alnk: qPrnConsole @ qstrlen
qPrnConsole:
	POP HL
	POP DE ; 'DЕ' a string kezd6cimére mutat
	PUSH DE
	PUSH HL
	LD B,D
	LD С,Е
	CALL STRLEN
	LD B,H
	LD C,L
	RST $30
	DB $22
	RET
;#end
```

(у цьому прикладі чітко видно, що є виклик CALL STRLEN, який зберігається в блоці 'qstrlen', тому для бездоганної роботи блок 'qstrlen' також повинен бути компонований)

`;alnk: 'procedure_name' [,'procedure_name2',... ] alias 'block_name'`

Може статися, що певні частини asm-програми, хоча вони мають кілька точок входу до C, все ще логічно або структурно пов'язані. Прикладом останнього є ситуація, коли після точки входу програма, що виконується, продовжується в частині, яка також має іншу точку входу. У цьому випадку можна використовувати наведену вище формулу.

Ми надаємо пов'язаному блоку окреме ім'я блоку та посилаємося на нього за допомогою слова "псевдонім" в кінці рядка "alnk:", що вказує на точку входу. Якщо пов'язаний блок має кілька точок входу C, тоді для кожної точки входу має бути визначений блок, і кожен блок посилається на те саме "ім'я_блоку" - звичайно, після "псевдоніма".

Наприклад:

```
;alnk: qDbGetSpriteAddr alias GET_SPRITE_ADDR

;alnk: GetSpriteDim alias GET_SPRITE_ADDR

;alnk: GET_SPRITE_ADDR
	qDbGetSpriteaddr
		pop bc
		pop hl
		ld a,l
		push hl
		push bc

	GetSpriteAddr
		ld hl,(qGfxBaseAddr)
		ld de,6
		add hl,de
		ld e,(hl)
		ld d,0
		inc hl
		ld b,h
		ld c,l
		ex de,hl
		add hl,hl
		add hl,de
		push hl
		
		ld e,c
		ld d,b
		ld l,a
		ld h,0
		add hl,hl
		add hl,de
		ld e,(hl)
		inc hl
		ld d,(hl)
		pop hl
		add hl,de
		ret

	GetSpriteDim
		call GetSpriteAddr
		ld a,(hl)
		inc hl
		ld b,(hl)
		dec hl
		ret
;#end
```

Зрозуміло, що блок має дві точки входу:
'qDbGetSpriteAddr' для коду програми на C, але також є 'GetSpriteAddr' для будь-яких внутрішніх викликів asm. Весь цей блок об'єднаний з іменем блоку 'GET_SPRITE_ADDR', і обидві точки входу оголошуються окремо перед блоком, щоб обидві посилалися на ім'я блоку 'GET_SPRITE_ADDR'.

Звичайно, рядок 'alnk: GET_SPRITE_ADDR' можна було б розширити посиланнями на зовнішні виклики - якби такі були в цьому розділі. У цьому випадку список посилань, позначений знаком '@', про який йшлося в попередньому пункті, можна використовувати після імені блоку.

Приклад такого визначення:

`;alnk: GET_SPRITE_ADDR @ qGetSpriteOffset, qGetParam`

Ще не згадувалося, що в посиланнях, що містять 'псевдонім' (і тільки тоді!!), назви процедур також можна перераховувати, розділяючи комами. У цьому випадку кожна назва процедури має, серед іншого, block_name.

Наприклад:

```
;alnk: CCCDR,CCCAR,CCGINT alias CCCDR_CCCAR_CCGINT

;alnk: CCCDR_CCCAR_CCGINT
CCCDR: inc hl ;Fetch integer from (HL+2)
	inc hl
			;Fetch a full 16-bit integer from the address in HL
CCCAR:
CCGINT: ld a,(hl)
	inc hl
	ld h,(hl)
	ld l,a
	ret
;#end
```

Після lib бажано також надати іншу інформацію, пов'язану з процедурами (наприклад, простий опис, використання, експерт тощо - див. нижче!), і ці посилання повинні йти після команди 'alnk:' без розривів рядків, тому бажано писати кожне посилання 'procedure name' alias 'block_name' на окремому рядку.

`;#end`

Під час компонування це позначає кінець розділу, який потрібно скопіювати у згенерований asm-код. Його потрібно вказувати після кожної команди ;alnk:', яка вказує на копіювання у фактично згенерований asm-код (тобто не містить 'alias').

Безпосередньо під посиланнями «alnk» можуть бути чотири різні команди:

|                                     |                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `;rem: 'опис'`                      | Можна написати будь-який опис. Будь-яку команду «rem:» можна розмістити одну під одною, але кожна з них вказує на текст, який відображається на окремому рядку. Важливо, щоб перший рядок завжди знаходився у верхньому рядку вкладки «Опис» панелі «Функції автоматичного компонування», а інші рядки розміщувалися під абзацом «Використання:». Важливо знати, що в описі можна використовувати знак `\|`, який також генерує розрив рядка. |
| `;use: 'використання'`              | Один рядок, що описує використання функції.                                                                                                                                                                                                                                                                                                                                                                                                   |
| `;aut: 'творець'`                   | Примітка, що стосується творця/експерта.                                                                                                                                                                                                                                                                                                                                                                                                      |
| `;lst: 'ім'я_розділу_документації'` | Іноді кілька інструкцій мають спільний, більш загальний опис. Їх можна написати окремо в так званому розділі документації. Команда 'lst:' відображає опис, позначений параметром 'doc_section_name'.                                                                                                                                                                                                                                          |

Наприклад:
```
;alnk: qDbInitGameLib alias DB_INIT
;rem: Inicializalja a GameLib-et.||vidMode = video mod (0 = 2 szin, 1 = 4 szin, 2 = 16 szin)
;rem: gameObjects = а GOBJ tomb kezdocime.|gameObjCount = GOBJ tomb elemeinek szama.
;rem: |Beallitja а grafikus modot, mind а négy lapra RAM-ot lapoz.
;rem: A rendert а hattérbufferre, а viewport-ot а teljes képernyore allitja.
;rem: DbInitGameLib (char vidMode, struct GOBJ *gameObjects, int gameObjCount);
;aut: Doberdo Brothers
;lst: GobjStruct

;alnk: DB_INIT @ qDbSetDefViewPort,qDbInitRender !
qDbInitGameLib
	pop ix
	pop de ; 'DE' = a GOBJ tomb mérete
	pop hl ; 'HL' = a GOBJ tomb cime
	pop bc ; 'BC' = Video mod
	push bc
	push hl
	push de
	push ix
	
	ld a,(3)             ; Alар memoria lapozas
	ld (SavedMemPages),a ; Elmentjuk
	ld a,($0B13)         ; Alap video mod
	ld (SavedVidMode),a
	and $fc
	or c
	out (6),а
	ret
;#end
```

`;doc: 'doc_section_name'`

Ця команда створює так званий розділ документа із заданим ім'ям. Це може бути, наприклад, загальний опис, що належить кільком записам (див. вище!).

Після цього рядка будь-яку команду 'rem:' можна написати безпосередньо після рядка без розриву рядка.

`;def: 'визначення'`

За допомогою цієї команди скрипта ми можемо по суті помістити визначення в нашу програму, синтаксис якого відповідає директиві \#define. (ніби ми просто замінили команду ';def:' словом '#define'.)

Наприклад:

```
;def: PAL_BLACK       0    // fekete paletta szin     
;def: PAL_DARKBLUE    1    // sotét kék paletta szin 
;def: PAL_DARKRED     4    // sotét piros paletta szin  
;def: PAL_DARKPURPLE  5    // sotét lila paletta szin    
;def: PAL_DARKGREEN   16   // sotét zold paletta szin  
;def: PAL_DARKCYAN    17   // sotét cian paletta szin 
;def: PAL_DARKYELLOW  20   // sotét sarga paletta szin
;def: PAL_GRAY        21   // szurke paletta szin    
;def: PAL_BLACK2      64   // fekete2 paletta szin    
;def: PAL_BLUE        65   // kék paletta szin        
;def: PAL_RED         68   // piros paletta szin      
;def: PAL_PURPLE      69   // lila paletta szin       
;def: PAL_GREEN       80   // zold paletta szin         
;def: PAL_CYAN        81   // cian paletta szin          
;def: PAL_YELLOW      84   // sarga paletta szin           
;def: PAL_WHITE       85   // fehér paletta szin           
```
